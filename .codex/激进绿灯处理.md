# 激进绿灯处理计划

## 目标

在 `压缩相邻消息` 的“处理世界书条目”模式下新增 `激进处理绿灯缓存`。开启后，符合条件的绿灯条目第一次触发时会被固定到真实聊天楼层，后续生成不再依赖绿灯触发条件，而是从聊天变量缓存中注入到同一楼层后方，再交给现有“处理聊天历史”逻辑合并，从而提高提示词缓存命中稳定性。

## 已确认规格

- 只处理真绿灯：`!entry.constant && !entry.vectorized`。
- 只固定非动态宏绿灯；动态宏判断沿用当前 `hasDynamicPromptMacro`，其中 `<USER>`、`<BOT>`、`<CHAR>`、`{{user}}`、`{{char}}` 不算动态宏。
- 固定锚点使用真实楼层和 swipe：`message_id + swipe_id`。
- 新固定绿灯锚定到最新未隐藏楼层，无论该楼层身份；若没有任何消息，则视为插入到所有消息前。
- 隐藏只看 `is_hidden`；system 消息不算隐藏。
- 固定绿灯全部以 `system` role 注入。
- 固定绿灯不要求存在 `{{压缩相邻消息::lora_key}}` 占位符。
- 已固定且 hash 一致的绿灯再次触发时，固定锚点胜出，并删除原世界书位置的重复内容。
- 始终只有一份绿灯内容：
  - 满足固定条件：固定到楼层缓存。
  - 含动态宏：按旧机制提取到 `lora_key`。
- 固定后使用内容快照注入；hash 校验使用世界书原始 content 的 hash。
- 世界书条目身份以 `world + uid` 为主，`name/comment` 仅作为显示和回退辅助；`uid` 仅在单本世界书内唯一。
- 若世界书条目 hash 不一致，不使用旧缓存；该条目若本轮重新触发，则作为新绿灯处理。
- 若楼层或 swipe 被删除，清除对应缓存。
- 若缓存锚点楼层存在但本轮未进入 prompt/无法匹配，则本轮不注入但保留缓存；如果同条目本轮作为新绿灯触发，则固定到新的楼层位置。
- 若遇到多个可用匹配的同一绿灯，以最先固定的为准。
- 关闭选项时不注入、不新增、不清理；缓存保留。
- UI：
  - `激进处理绿灯缓存` checkbox 放在“提取无动态宏的蓝灯持久条目到 ...”之前。
  - 提供“清空绿灯缓存”按钮，清理当前聊天变量 `压缩相邻消息.绿灯缓存`。

## 数据结构

聊天变量路径：

```ts
压缩相邻消息.绿灯缓存
```

建议结构：

```ts
{
  version: 1,
  entries: [
    {
      world: string,
      uid: number,
      name: string,
      content_hash: string,
      content_snapshot: string,
      fixed_at: {
        message_id: number | null,
        swipe_id: number | null,
        message_hash: string | null
      },
      created_at: number
    }
  ]
}
```

其中 `message_id: null` 表示插入到所有消息前。

## 实现顺序

1. 扩展设置：
   - `settings.entry_processing.worldbook.aggressive_green_cache.enabled`
   - 默认 `false`。
2. 扩展 UI：
   - 在世界书处理区块最前方加入 checkbox。
   - 加入清空按钮，操作当前聊天变量缓存。
3. 缓存辅助函数：
   - 读取/写入/清空 `压缩相邻消息.绿灯缓存`。
   - 计算稳定 content hash。
   - 校验世界书条目是否仍存在且 hash 一致。
   - 清除楼层或 swipe 已不存在的缓存。
4. prompt 楼层映射：
   - 用 `getChatMessages('0-{{lastMessageId}}', { hide_state: 'unhidden' })` 获取真实未隐藏楼层。
   - 在分割出的聊天历史 chunk 中按顺序保守匹配 message 内容。
   - 匹配不到则本轮跳过注入，不猜位置。
5. 绿灯处理流程：
   - 先消费原世界书位置，再插入固定绿灯，避免误删新插入内容。
   - 已固定且可用：消费原位置重复项，稍后插入固定锚点。
   - 未固定且满足条件：消费原位置，写入缓存，稍后插入最新未隐藏楼层后。
   - 动态宏绿灯：继续走旧 `lora_key` 提取。
6. 插入固定绿灯：
   - 统一插入 `role: 'system'`。
   - 插入发生在聊天历史 squash 之前。
7. 验证：
   - lint `src/酒馆助手/压缩相邻消息/squash.ts`。
   - `npm run build`。
   - 确认只改压缩相邻消息相关源码与 dist，避免触碰格式补全脚本。

## 中止条件

允许中止编码并汇报：

- 无法可靠把 prompt 中的聊天消息映射回真实楼层，且继续实现会导致固定绿灯插错位置。
- SillyTavern 事件数据无法区分本轮绿灯和历史缓存，导致无法保证“始终只有一份”。
- 需要复刻 SillyTavern 世界书扫描逻辑才能满足需求。
